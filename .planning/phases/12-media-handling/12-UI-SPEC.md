# Phase 12 — UI Design Contract

> Visual and interaction contract for Phase 12: Media Handling.
> Generated by gsd-ui-researcher, verified by gsd-ui-checker.

---

## Scope Note

Phase 12 is a backend-only phase. It adds one new Edge Function (`whatsapp-media`) and patches
`whatsapp-webhook` to route inbound media messages. There are **no new mobile app screens** and
**no new React Native components** in this phase.

The only "UI surface" this phase owns is the **WhatsApp bot reply text strings** sent back to the
user via `whatsapp-send`. These are defined in the Copywriting Contract below and are the primary
deliverable of this spec.

All design tokens below reflect the existing project design system and are pre-populated from
`constants/theme.ts`. They are provided as reference so the executor understands the context, but
no new styled components will be built.

## Copywriting Contract

Phase 12 introduces no new in-app copy. The copywriting contract below governs the **WhatsApp bot
reply messages** sent via `whatsapp-send`. These are the entire user-facing output of this phase.

Reply messages must use plain text (WhatsApp does not render markdown). Tone: concise,
confirmation-first, action-oriented when the user needs to retry.

| Element | Copy |
|---------|------|
| Primary CTA (payment proof success) | "Got it! I've attached your payment photo as proof for {monthName} {year}. Your payment is now recorded as "{newStatus}"." |
| Primary CTA (document success) | "Your document "{filename}" has been saved to your property files. You can view it in the Dwella app under Documents." |
| Empty state — no tenant found | "I couldn't find a linked tenant account for your number. Link your account first in the Dwella app." |
| Classification failure (image not recognized) | "I couldn't tell if this photo is a payment receipt. If it is, add a caption like "payment for March" and send again, or log the payment via text: "Mark Ravi's March rent as paid"" |
| Error — processing failed (generic) | "Something went wrong processing your media. Please try again or use the Dwella app directly." |
| Error — unsupported media type | "I can only accept photos (payment proofs) and documents (PDFs, Word files). For other requests, just type your message." |
| Ambiguous tenant (multiple tenancies) | "You're linked to multiple properties. Which property is this for — {property1} or {property2}?" |
| Destructive action | None in this phase. Media attachment is non-destructive (upsert with overwrite). |

Copy rules:
- Never exceed 3 sentences per reply.
- Start success replies with "Got it!" or the completed action (confirmation-first).
- Start failure replies with the problem, then the solution in the same message.
- Use the user's own words when echoing back (filename, month name, property name).
- Do not use exclamation marks in error/failure replies.

## Interaction States

These states apply to the WhatsApp bot reply flow, which is the only interaction surface in Phase 12.

| Trigger | Bot State | Reply Sent |
|---------|-----------|-----------|
| Inbound image, tenant identified, classification = payment proof | Success | Payment proof attached confirmation |
| Inbound image, tenant identified, classification = not payment proof | Failure | Classification failure message |
| Inbound image, no linked user found | Error | No tenant found message |
| Inbound image, multiple tenancies, current-month heuristic resolves | Auto-resolved | Payment proof attached confirmation (no user prompt) |
| Inbound image, multiple tenancies, ambiguous after heuristic | Clarification needed | Ambiguous tenant message |
| Inbound document, tenant/user identified | Success | Document saved confirmation |
| Inbound document, no linked user found | Error | No tenant found message |
| Inbound media, unsupported type (video, sticker, audio) | Error | Unsupported media type message |
| Processing error (CDN download failure, Storage failure, Claude API failure) | Error | Generic processing failed message |

Processing indicator: WhatsApp does not support typing indicators from the Cloud API in all cases.
No "processing..." intermediate message is sent — the reply is the only message. Latency target is
under 10 seconds from webhook receipt to reply delivery.
